# SMART-SFCS용 Gemini 3.1 메인 프롬프터 가이드

## 1) 목적
이 문서는 SMART-SFCS를 Gemini 3.1으로 분석할 때,
- 프로젝트 구조를 빠르게 파악하고
- 역할 간 소통 품질을 높이며
- 추가 카테고리(분류 체계)를 일관되게 설계하기 위한 기준을 제공합니다.

---

## 2) 현재 프로그램 분석 요약 (프롬프트 입력 전제)
- 핵심 도메인: 건설 골조 공정 관제
- 주요 기능: 동/층/호 공정 상태 관리, AI 도면 분석, 실시간 채팅, 권한 기반 운영
- 주요 역할: 제작자(CREATOR), 관리자(ADMIN), 작업자(WORKER), 협력사(SUBCONTRACTOR)
- AI 분석 출력 핵심 구조:
  - `buildingStructures` (동별 층수/세대/죽은세대 로직)
  - `riskFactors` (카테고리 기반 위험요인)
  - `actionItems` (실행 항목)

## 3) Gemini 3.1 메인 프롬프터 (복붙용)
아래 텍스트를 그대로 Gemini 3.1에 입력하세요.

```text
[System]
당신은 건설 현장 디지털 관제 시스템 분석가이자 정보구조(IA) 설계자다.
목표는 SMART-SFCS 코드/기능을 분석하여, 역할 간 소통 효율을 높이는 카테고리 체계를 설계하는 것이다.
반드시 한국어로 답변하고, 모호한 판단은 가정(Assumption)을 먼저 명시한다.
출력은 JSON과 표 형식을 함께 제공한다.

[User]
다음 프로그램을 분석해줘.

프로그램 성격:
- 건설 골조 공정 실시간 관제
- 권한별 UI/기능 분리 (제작자/관리자/작업자/협력사)
- AI 도면 분석 결과를 현장 운영 데이터와 동기화
- 실시간 채팅으로 현장 소통 수행

분석 목표:
1) 현재 카테고리(공정, 안전, 리스크, 액션)의 부족한 점 진단
2) 소통 강화를 위한 신규 카테고리 제안
3) 각 카테고리별 필수 데이터 필드 정의
4) UI에 표시할 우선순위(상/중/하)와 알림 트리거 제안
5) 카테고리 충돌/중복 방지 규칙 제안

반드시 포함할 제약:
- 현장 사용자가 5초 안에 이해 가능한 라벨
- 관리자 승인 흐름(승인요청/승인완료/반려)과 연결 가능
- 기존 데이터 구조(buildingStructures, riskFactors, actionItems)와 호환
- 카테고리 수는 6~10개 이내

원하는 출력 형식:
A. 요약(5줄 이내)
B. "신규 카테고리 제안표"(카테고리명, 목적, 입력데이터, 생성주체, 소비주체, 알림조건, KPI)
C. "카테고리 스키마 JSON" 예시
D. "도입 우선순위 로드맵"(즉시/1개월/분기)
E. "실패 시나리오 및 보정 룰" 3개
```

## 4) 추가 카테고리 작성 기준 (Checklist)
신규 카테고리를 만들 때 아래 항목을 모두 채우면 품질이 안정됩니다.

1. 카테고리명
- 명확한 업무 단위 명사 사용 (예: 승인지연, 자재이슈, 안전경고)

2. 운영 목적
- “무엇을 더 빨리/정확하게 의사결정하게 하는지” 1문장

3. 데이터 필드
- 최소 필드: `id`, `category`, `title`, `detail`, `severity`, `ownerRole`, `targetRole`, `building`, `floor`, `unit`, `timestamp`, `status`

4. 이벤트 트리거
- 어떤 조건에서 생성되는지 정량 조건 명시
- 예: 승인요청 후 2시간 미처리 시 `승인지연` 생성

5. 액션 연결
- `actionItems`로 자동 변환 가능한지 여부
- 담당자/기한(`assignee`, `dueAt`) 포함

6. 노출 우선순위
- 대시보드 상단 고정 여부
- 채팅/알림/현황판 중 어디에 먼저 노출할지

7. 중복 방지 규칙
- 동일 동/층/호 + 동일 카테고리 + 동일 원인코드일 때 병합

8. 종료 조건
- 어떤 상태 변경 시 닫힘 처리되는지 명확히 정의

## 5) 소통 강화를 위한 추천 신규 카테고리 (초안)
아래 8개는 현재 SFCS 기능과 연결이 쉬운 기본안입니다.

1. 승인 지연
- 승인요청 이후 임계시간 초과 건 추적

2. 반려 사유 클러스터
- 반려 사유를 표준 코드화하여 재발 방지

3. 공정-채팅 불일치
- 채팅 보고 내용과 실제 상태값 불일치 탐지

4. 안전 경고 우선순위
- 고위험 안전요인 자동 상향

5. 자재/장비 병목
- 작업 지연 원인이 자재/장비인지 분리

6. 협력사 응답 지연
- 협력사 회신 SLA 초과 이슈 분류

7. 재작업(리워크) 발생
- 동일 위치 반복 작업 감지

8. 데드유닛(Dead Unit) 예외
- 도면 규칙과 현장 입력이 충돌하는 케이스 추적

## 7) 실제 적용 순서 (권장)
1. Gemini 3.1으로 카테고리 초안 생성
2. 제작자/관리자 1차 검토 (라벨 이해도, 알림 피로도)
3. `riskFactors` 매핑 규칙 확정
4. `actionItems` 자동 생성 규칙 연결
5. 1주 파일럿 후 중복/오탐 보정

이 순서를 따르면 기능 추가 없이도 운영 언어(소통 체계)부터 빠르게 표준화할 수 있습니다.

## 8) 역할별 소통 템플릿 (운영 적용)
고우선(`priority=high`) 실행 항목이 발생했을 때, 같은 이벤트를 역할별 문장으로 분기해 전파합니다.

1. 관리자용
- `[관리자 지시] {현장명} - 긴급 항목 {N}건 확인 후 담당 지정 및 기한 설정: {핵심 2건}`

2. 작업자용
- `[작업자 안내] {현장명} - 긴급 항목 {N}건 관련 즉시 점검 및 진행상태 보고: {핵심 2건}`

3. 협력사용
- `[협력사 요청] {현장명} - 긴급 항목 {N}건 대응 계획/회신 필요: {핵심 2건}`

4. 통합 브리핑용
- `[긴급 액션 공유]`
  - 관리자: 담당 배정 및 승인 경로 점검
  - 작업자: 즉시 현장 점검 및 보고
  - 협력사: 대응 계획 회신
  - 핵심: `{핵심 2건}`

권장 규칙:
- 동일 분석 키(현장명+프로젝트코드+액션 해시)에서는 중복 전송하지 않습니다.
- 채팅 전송 순서는 관리자 → 작업자 → 협력사 → 통합 브리핑 순으로 고정합니다.

## 9) 2026-03-07 업그레이드/디자인 통합 점검 (최신)
아래 블록은 "현재까지 업그레이드 상태 + 디자인 전체 반영사항"을 기준으로 Gemini 점검을 실행할 때 사용합니다.

3. AI 분석 파이프라인

- JSON 백업/복구, 제작자 배치 제어(초기화/재동기화)

## 10) 마스터 통합본 (업그레이드 상세 + 디자인 전체)
기준일: 2026-03-07

### 10-1. 프로젝트 상태 요약
- 제품명: SMART-SFCS (스마트 골조 통합 관제)
- 프론트엔드: React 19 + TypeScript + Vite
- 시각화/아이콘: Recharts, Lucide
- AI: Google `@google/genai` (`gemini-3-flash-preview`)
- 실시간 데이터: Firebase Firestore + 익명 인증 + IndexedDB persistence
- PTW 아카이브: Supabase (`gangform_ptw_records`)

### 10-2. 업그레이드 완료 상세
#### A) 역할 기반 운영 고도화
- 역할 체계: 제작자(CREATOR), 관리자(ADMIN), 작업자(WORKER), 협력사(SUBCONTRACTOR)
- 권한 분리:
  - 제작자: AI 업로드/재분석, DB 초기화/재동기화, 시스템 제어
  - 관리자: 승인/반려, 백업/복구, 채팅 관리
  - 작업자: 상태 진행/승인요청, PTW 작성/증빙 업로드
  - 협력사: 운영 소통 대상(역할별 전파 메시지)
- 관리자/제작자 로그인 모달(코드 분기) 기반 권한 전환

#### B) 대시보드/공정 상태 플로우
- 상태 체계: `미착수 → 설치중 → 승인요청 → 승인완료 → 타설중 → 양생완료`
- 작업자/관리자 상태 전이 분리
- 승인요청 긴급 카드, 즉시 승인 액션, 동/층 점프 지원
- 승인 반려 기능 추가(관리자/제작자)
- SiteMap + 동별 카드 실시간 진행층 표시
- Dead Unit 제외 계산 적용

#### C) AI 도면/데이터 분석 엔진
- 입력: 이미지/PDF/엑셀(CSV 포함) 멀티 업로드
- 전처리: 이미지 압축, MIME 정규화, 진행률 표시
- 핵심 분석: 동별 독립 구조, 층/호수 추출, Dead Unit 규칙 추론
- 출력 표준: `buildingStructures`, `riskFactors`, `actionItems`
- 후처리: 리스크 코드/심각도 매핑, 액션 코드/우선순위 정규화
- 결과 저장/구독: Firestore 영속화 + 실시간 구독
- 권한 제어: 제작자만 재분석, 타 역할 조회 전용

#### D) 소통/알림 체계
- Live Chat 실시간 채팅(역할별 뱃지/버블 분기)
- 관리자/제작자 채팅 전체 삭제 기능
- 시스템 토스트 + 브라우저 알림 + 사운드
- 고우선 액션 발생 시 역할별 커뮤니케이션 번들 전송
- 카카오톡/문자 공유를 위한 메시지 공유/클립보드 흐름

#### E) PTW 모듈
- PTW 탭/페이지/히스토리 분리
- Worker:
  - 필수 체크(압축강도, TBM/PPE, 하부통제, 낙하물 제거)
  - 작업 전 4장 + 작업 중 1장 증빙 구조
  - 승인요청/완료, 연습모드, 되돌리기, 로컬 임시저장
- Admin:
  - 제출 검토 + 승인/반려
- 상태: `draft → requested → approved → completed / rejected`
- 저장: Firestore `site_data/gangform_ptw`
- 아카이브: Supabase insert + 히스토리 조회

#### F) 운영 안정성
- 전체 상태 JSON 백업/복구
- 제작자 배치 동작(초기화/재동기화)
- 연결 상태(online/offline) 및 persistence 반영
